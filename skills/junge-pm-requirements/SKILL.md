# 俊哥 - 项目管理要求 Skill

**Skill 名称**: `junge-pm-requirements`  
**版本**: v1.0  
**创建日期**: 2026-03-15  
**适用范围**: 所有服务俊哥的 AI 团队成员

---

## 🎯 核心原则

### 1. AI 速度评估（最重要）
**俊哥明确要求**: 所有工作量评估按 **AI 团队速度** 计算，**不是人类工时**！

| 任务类型 | 人类工时 | AI 工时 | 倍数 |
|----------|---------|--------|------|
| 代码开发 | 8 小时/功能 | 15-30 分钟/功能 | 16-32 倍 |
| Bug 修复 | 1 小时/Bug | 3-5 分钟/Bug | 12-20 倍 |
| 文档编写 | 2 小时/文档 | 5-10 分钟/文档 | 12-24 倍 |
| 测试执行 | 1 小时/套件 | 3-5 分钟/套件 | 12-20 倍 |
| 代码审查 | 2 小时/PR | 5 分钟/PR | 24 倍 |
| GitHub 同步 | 30 分钟 | 2-5 分钟 | 6-15 倍 |

**错误示例** ❌:
> "预计 3-4 周完成开发"（人类工时）

**正确示例** ✅:
> "预计 30-40 分钟完成开发"（AI 工时）

---

### 2. 进度汇报频率
- **定时汇报**: 每 **30 分钟** 向俊哥汇报一次进度
- **汇报内容**: 各团队状态 + 总体进度表 + 本周期完成事项 + 下一周期计划 + 需要决策事项
- **格式要求**: 用表格和 emoji 让进度一目了然

**汇报模板**:
```markdown
## 📋 项目进度看板 - [项目名称]

| 任务 | 负责人 | 状态 | 进度 | 更新时间 | 备注 |
|------|--------|------|------|----------|------|
| XX 设计 | product-ux | ✅ 完成 | 100% | 19:00 | 已输出原型 |
| XX 开发 | fullstack | 🔄 进行中 | 60% | 19:05 | 等待确认 XX |

### 待俊哥确认事项
- [ ] XX 问题（负责人：XXX，紧急程度：高/中/低）

### 下一步计划
- XXX 预计 XX 时间完成
```

---

### 3. 文档管理要求

#### 3.1 PRD 文档必须实时更新
**要求**: 任何设计、架构、技术选型变化都要**立即更新 PRD 文档**

**更新内容**:
- ✅ 技术选型变更（数据库/OSS/框架等）
- ✅ 架构评审结果（6 大维度评分 + 风险清单）
- ✅ Bug 修复记录（P0/P1/P2 分级）
- ✅ 测试结果（完整功能测试 + 截图索引）
- ✅ GitHub 同步记录

**错误示例** ❌:
> PRD 文档是旧的，技术选型和实际代码不一致

**正确示例** ✅:
> PRD 文档始终是最新的，包含所有变更和技术选型

#### 3.2 SOP 文档合并
**要求**: 所有流程文档合并到**一个 MD 文件**，不要分多个文档

**文档结构**:
```markdown
# 学习助手项目 - 完整 SOP

## 1. 团队角色与职责
## 2. 工作流程
## 3. 质量审计结果
## 4. 文档清单
## 5. GitHub 同步记录
```

#### 3.3 文档位置规范
```
项目根目录/
├── docs/
│   ├── PROJECT-SOP.md           # 完整 SOP（合并后）
│   ├── test-reports/            # 测试报告
│   ├── screenshots/             # 测试截图
│   └── bug-fix-log.md           # Bug 修复日志
├── project/v1-prd/docs/
│   ├── PRD_v1.1.md              # PRD 主文档（实时更新）
│   └── PRD_CHANGELOG.md         # PRD 变更记录
└── memory/
    └── 2026-03-15.md            # 项目关键记忆
```

### 4. 测试覆盖要求

#### 4.1 测试必须覆盖所有场景
**错误示例** ❌:
> 只测试了登录页面，没有测试其他功能页面

**正确示例** ✅:
> 测试了所有 21 个场景：
> - 未登录场景（5 个页面）
> - 登录场景（3 个）
> - 注册场景（3 个）
> - 登录后功能场景（6 个页面）
> - 边界场景（4 个）

#### 4.2 截图必须包含所有场景
**要求**: 每个测试场景都要有截图，俊哥要看到真实的测试过程

**截图清单**:
- ✅ 未登录页面（5 张）
- ✅ 登录流程（3 张）
- ✅ 注册流程（3 张）
- ✅ 登录后功能（6 张）
- ✅ 边界场景（4 张）
- ✅ **总计 21+ 张**

#### 4.3 浏览器测试必须真实操作
**要求**: 俊哥要看到浏览器真实操作，不是模拟测试

**测试内容**:
- ✅ 打开浏览器访问页面
- ✅ 真实输入（手机号/验证码）
- ✅ 真实点击（登录/注册/提问）
- ✅ 真实验证（AI 回复/页面跳转）
- ✅ 每步截图保存

---

### 5. GitHub 同步要求

#### 5.1 同步频率
- **自动同步**: 每 **30 分钟** 自动推送到 GitHub
- **手动同步**: 重要文档更新后立即推送

#### 5.2 同步内容
**会同步**:
- ✅ 所有源代码（backend/ + frontend/ + mobile/）
- ✅ 所有文档（docs/ + project/v1-prd/docs/）
- ✅ 配置文件（.env.example, package.json, prisma/schema.prisma）
- ✅ 测试报告和截图

**不同步** (.gitignore):
- ❌ node_modules/
- ❌ .env（敏感配置）
- ❌ *.db（本地数据库）
- ❌ uploads/（用户上传文件）
- ❌ 日志文件

#### 5.3 Commit 规范
```bash
git commit -m "Initial commit: 学习助手项目完整代码"
git commit -m "feat: 添加完整功能测试（21 个场景）"
git commit -m "docs: 更新 PRD 文档（包含实施记录）"
```

### 6. 团队角色与职责

#### 6.1 完整团队（6 人）
1. **项目经理 (PM)** - 统筹协调、任务分配、进度跟踪、决策汇报（每 30 分钟向俊哥汇报）
2. **产品设计师 (Product-UX)** - 需求分析、产品设计、原型设计、PRD 输出
3. **架构师 (Architect)** - 6 大架构设计 + 评审 + 风险识别
4. **项目助理** - 文档审查、过程审计、GitHub 同步（每 30 分钟）、质量把控、变更管理
5. **fullstack** - 后端/前端开发、数据库设计、单元测试
6. **qa** - API 测试、前端测试、浏览器 E2E、测试报告、Bug 跟踪

#### 6.2 工作流程（4 阶段）
```
阶段 1: 需求分析与设计 → 产品设计师 → 架构师 → 项目经理
阶段 2: 开发实施 → fullstack → 项目助理
阶段 3: 测试验证 → qa → fullstack(Bug 修复) → 项目助理
阶段 4: 交付归档 → 项目助理 → 项目经理 → 俊哥
```

---

### 7. 质量门禁

| 阶段 | 检查项 | 通过标准 |
|------|--------|----------|
| **设计阶段** | PRD 评审 | 功能完整 + 技术可行 |
| **开发阶段** | 代码审查 | 无 P0/P1 问题 |
| **测试阶段** | 测试报告 | 通过率>95% + 全场景覆盖 |
| **交付阶段** | 质量审计 | 文档完整 + 一致 + PRD 最新 |

---

### 8. Bug 分级标准

| 级别 | 定义 | 修复时限（AI 速度） |
|------|------|------------------|
| **P0** | 严重 Bug，阻塞核心功能 | 5-10 分钟 |
| **P1** | 重要 Bug，影响主要功能 | 15-30 分钟 |
| **P2** | 轻微 Bug，不影响核心功能 | 1 小时 |
| **P3** | 建议改进 | 后续迭代 |
