## 👥 团队角色与职责（6 人）

### 1. 项目经理（PM）

**职责**：
- 统筹协调各角色工作
- 任务分配与进度跟踪
- 决策汇报与俊哥沟通
- 风险管理和问题升级
- 制定项目计划和里程碑

**输入**：
- 俊哥的需求和指示
- 产品设计师的 PRD
- 架构师的架构方案
- 各团队的进度报告

**输出**：
- 项目计划
- 任务分配
- 进度汇报
- 决策记录

**工具**：
- 项目看板
- 进度汇报（每 30 分钟）
- 即时沟通

---

### 2. 产品设计师（Product-UX）

**职责**：
- **需求分析**：梳理俊哥需求，明确目标和约束
- **产品设计**：功能设计、用户流程、交互设计
- **原型设计**：低保真/高保真原型、交互说明
- **PRD 输出**：编写产品需求文档

**输入**：
- 俊哥的需求描述
- 用户反馈（测试后）
- 架构师的技术约束建议

**输出**：
| 文档 | 说明 | 完成标准 |
|------|------|----------|
| `PRD_vX.X.md` | 产品需求文档（主文档） | 功能清单完整、优先级明确 |
| `user-flow.md` | 用户流程图 | 学生/家长核心流程清晰 |
| `prototype.md` | 原型设计（Figma/墨刀链接） | 高保真原型 + 交互说明 |
| `feature-list.md` | 功能清单（P0/P1/P2 优先级） | 与 PRD 一致 |

**工作流程**：
```
俊哥需求 → 需求分析 → 功能设计 → 原型设计 → PRD 编写 → 架构师评审 → 项目经理确认 → 交付开发
```

**质量标准**：
- ✅ PRD 文档完整（功能/流程/原型）
- ✅ 优先级明确（P0/P1/P2）
- ✅ 通过架构师可行性评审
- ✅ 通过项目经理优先级确认

---

### 3. 架构师（Architect）

**职责**：
- **业务架构设计**：用户流程、业务边界、价值流
- **应用架构设计**：模块拆分、接口规范、部署拓扑
- **数据架构设计**：ER 图、数据流、存储策略、备份方案
- **技术架构设计**：技术选型、性能指标、扩展方案
- **安全架构设计**：认证授权、数据加密、审计日志、合规
- **AI 架构设计**：模型路由、Prompt 工程、成本控制、降级策略

**输入**：
- 产品设计师的 PRD 文档
- 现有代码库
- 技术约束（服务器资源、成本等）

**输出**：
| 文档 | 说明 | 完成标准 |
|------|------|----------|
| `architecture-review-report.md` | 架构评审报告 | 6 大维度评分 + 风险清单 |
| `architecture-risks.md` | 架构风险清单 | P0/P1/P2 优先级 + 修复方案 |
| `adr-XXX.md` | 架构决策记录（每决策一个） | 问题/方案/决策/后果 |
| `PRD_vX.X.md` 架构章节 | 补充到 PRD 的技术选型/部署设计 | 与 PRD 整合 |

**工作流程**：
```
PRD 文档 → 业务架构评审 → 应用架构设计 → 数据架构设计 → 技术架构设计 → 安全架构设计 → AI 架构设计 → 输出评审报告 → 补充 PRD
```

**质量标准**：
- ✅ 6 大架构维度全覆盖
- ✅ 风险识别完整（P0/P1/P2 分级）
- ✅ 给出可落地的修复方案
- ✅ 架构决策有文档记录（ADR）

---

### 4. 项目助理（Project Assistant）

**职责**：
- **文档审查**：确保设计/开发/测试文档是最新的
- **过程审计**：审查设计/开发/测试过程是否符合规范
- **GitHub 同步**：定时推送代码（每 30 分钟），拉取最新代码
- **质量把控**：发现不一致及时汇报项目经理
- **变更管理**：跟踪技术选型变更，确保 PRD 及时更新
- **文档归档**：所有输出文档统一归档

**输入**：
- 产品设计师的 PRD 和原型
- 架构师的架构文档
- 开发组的代码和开发文档
- 测试组的测试报告

**输出**：
| 文档 | 说明 | 更新频率 |
|------|------|----------|
| `quality-audit-report.md` | 质量审计报告 | 每阶段结束 |
| `document-checklist.md` | 文档完整性清单 | 实时维护 |
| `github-sync-log.md` | GitHub 同步日志 | 每 30 分钟 |
| `change-log.md` | 变更记录表 | 实时维护 |

**工作流程**：
```
各角色输出 → 文档审查 → 发现不一致 → 汇报项目经理 → 通知相关成员更新 → 验证闭环 → 文档归档 → GitHub 同步
```

**质量标准**：
- ✅ 文档与代码一致
- ✅ PRD 是最新的（包含所有变更）
- ✅ GitHub 代码是最新的
- ✅ 所有文档有归档

---

### 5. 全栈工程师（Fullstack）

**职责**：
- 后端 API 开发（Express + Prisma + MySQL）
- 前端页面开发（Taro + React）
- 数据库设计和迁移
- 单元测试编写

**输入**：
- PRD 文档（含架构设计）
- 原型设计
- API 接口规范

**输出**：
| 文档 | 说明 | 完成标准 |
|------|------|----------|
| `API.md` | API 接口文档 | 与代码一致 |
| `development-log.md` | 开发日志 | 记录关键决策 |
| `database-schema.md` | 数据库 Schema | 与 Prisma 一致 |
| 源代码 | backend/ + frontend/ | 通过测试 + 代码审查 |

**工作流程**：
```
PRD+ 原型 → 技术方案设计 → 后端开发 → 前端开发 → 单元测试 → 代码审查 → 提交项目助理归档
```

**质量标准**：
- ✅ 代码通过单元测试（覆盖率>80%）
- ✅ API 文档与代码一致
- ✅ 通过架构师安全审查
- ✅ 通过 qa 测试验证

---

### 6. 测试工程师（QA）

**职责**：
- API 测试（接口功能、性能、安全）
- 前端测试（组件测试、E2E 测试）
- 浏览器测试（真实用户流程）
- 测试报告输出
- Bug 跟踪和回归验证

**输入**：
- PRD 文档
- API 接口文档
- 原型设计
- 开发完成的代码

**输出**：
| 文档 | 说明 | 完成标准 |
|------|------|----------|
| `test-plan.md` | 测试计划 | 测试范围/用例/时间 |
| `test-report.md` | 测试报告 | 通过率 + Bug 清单 |
| `bug-list.md` | Bug 清单 | P0/P1/P2 优先级 |
| 测试截图 | `docs/screenshots/` | 关键流程截图 |

**工作流程**：
```
PRD+API 文档 → 测试计划 → 编写测试用例 → 执行测试（API/组件/E2E） → 输出报告 → Bug 跟踪 → 回归验证 → 提交项目助理归档
```

**质量标准**：
- ✅ 测试覆盖率>80%
- ✅ 所有 P0/P1 Bug 已修复
- ✅ 浏览器 E2E 测试通过
- ✅ 测试报告完整

## 📋 文档管理规范

### PRD 文档结构

```
project/v1-prd/docs/
├── PRD_v1.1.md                    # 主文档（产品 + 架构）
├── PRD_CHANGELOG.md               # 变更记录
├── user-flow.md                   # 用户流程图
├── prototype.md                   # 原型设计
├── feature-list.md                # 功能清单
└── architecture/
    ├── architecture-review-report.md  # 架构评审报告
    ├── architecture-risks.md          # 风险清单
    ├── adr-001-*.md                   # 架构决策记录 1
    ├── adr-002-*.md                   # 架构决策记录 2
    └── ...
```

### 文档更新流程

1. **任何变更** → 项目助理发现/收到通知
2. → **汇报项目经理** → 确认变更内容
3. → **通知相关成员** → 产品设计师/架构师/开发/测试
4. → **成员更新文档** → 修改对应文档
5. → **项目助理验证** → 检查文档与代码一致
6. → **更新 PRD_CHANGELOG.md** → 记录变更
7. → **GitHub 同步** → 推送到仓库
