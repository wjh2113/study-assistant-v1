## 1. 团队角色与职责

### 1.1 项目经理（PM）

**核心职责**：统筹协调、任务分配、进度跟踪、决策汇报

| 工作项 | 频率 | 输入 | 输出 |
|--------|------|------|------|
| 项目计划制定 | 项目启动 | 俊哥需求、PRD | 项目计划表 |
| 任务分配 | 每阶段开始 | PRD、架构方案 | 任务清单 |
| 进度跟踪 | 持续 | 各团队进度 | 进度看板 |
| 进度汇报 | **每 30 分钟** | 各团队进度 | 俊哥汇报 |
| 决策记录 | 按需 | 风险升级 | 决策记录 |
| 最终验收 | 项目结束 | 交付物 | 验收报告 |

**关键技能**：
- 项目管理（PMP/敏捷）
- 沟通协调
- 风险管理
- 决策能力

---

### 1.2 产品设计师（Product-UX）

**核心职责**：需求分析、产品设计、原型设计、PRD 输出

| 工作项 | 频率 | 输入 | 输出 |
|--------|------|------|------|
| 需求分析 | 项目启动 | 俊哥需求 | 需求清单 |
| 功能设计 | 设计阶段 | 需求清单 | 功能清单 |
| 用户流程设计 | 设计阶段 | 功能清单 | 用户流程图 |
| 原型设计 | 设计阶段 | 功能清单 | 原型（Figma/墨刀） |
| PRD 编写 | 设计阶段 | 以上所有 | PRD 文档 |
| 设计评审 | 评审会议 | 架构师意见 | PRD 修订版 |

**关键技能**：
- 产品设计
- 用户体验设计
- 原型工具（Figma/墨刀）
- 文档编写

---

### 1.3 架构师（Architect）

**核心职责**：6 大架构设计 + 评审 + 风险识别

| 架构维度 | 设计内容 | 输出文档 |
|----------|----------|----------|
| **业务架构** | 用户流程、业务边界、价值流 | 业务流程图 |
| **应用架构** | 模块拆分、接口规范、部署拓扑 | 架构拓扑图 |
| **数据架构** | ER 图、数据流、存储策略、备份方案 | 数据架构图 |
| **技术架构** | 技术选型、性能指标、扩展方案 | 技术选型报告 |
| **安全架构** | 认证授权、数据加密、审计日志、合规 | 安全评审报告 |
| **AI 架构** | 模型路由、Prompt 工程、成本控制、降级策略 | AI 架构方案 |

| 工作项 | 频率 | 输入 | 输出 |
|--------|------|------|------|
| PRD 评审 | 设计阶段结束 | PRD 文档 | 评审意见 |
| 6 大架构设计 | 设计阶段 | PRD+ 评审意见 | 架构文档 |
| 风险识别 | 持续 | 架构设计 | 风险清单 |
| 架构决策 | 按需 | 技术方案 | ADR 文档 |
| 补充 PRD | 设计阶段结束 | 架构文档 | PRD 架构章节 |

**关键技能**：
- 系统架构设计
- 技术选型
- 安全架构
- 风险管理

---

### 1.4 项目助理（Project Assistant）

**核心职责**：文档审查、过程审计、GitHub 同步、质量把控、变更管理、文档归档

| 工作项 | 频率 | 输入 | 输出 |
|--------|------|------|------|
| 文档审查 | 每阶段结束 | 各角色文档 | 审查报告 |
| 过程审计 | 每阶段结束 | 工作记录 | 审计报告 |
| GitHub 同步 | **每 30 分钟** | 代码变更 | 同步日志 |
| 质量把控 | 持续 | 文档/代码 | 质量问题清单 |
| 变更管理 | 持续 | 变更请求 | 变更记录 |
| 文档归档 | 每阶段结束 | 所有文档 | 归档清单 |

**关键技能**：
- 文档管理
- 质量管理
- Git/GitHub
- 细节导向

### 1.5 全栈工程师（Fullstack）

**核心职责**：后端开发、前端开发、数据库设计、单元测试

| 工作项 | 频率 | 输入 | 输出 |
|--------|------|------|------|
| 技术方案设计 | 开发阶段开始 | PRD+ 架构文档 | 技术方案 |
| 数据库设计 | 开发阶段 | PRD+ 架构文档 | 数据库 Schema |
| 后端开发 | 开发阶段 | PRD+API 规范 | 后端代码 |
| 前端开发 | 开发阶段 | PRD+ 原型 | 前端代码 |
| 单元测试 | 开发阶段 | 代码 | 测试用例 |
| 代码审查 | 提交前 | 自己代码 | 审查记录 |
| Bug 修复 | 测试阶段 | Bug 清单 | 修复代码 |

- 后端开发（Express/Node.js）

- 数据库（MySQL/Prisma）

| 工作项 | 频率 | 输入 | 输出 |

| API 测试 | 测试阶段 | API 文档 | API 测试报告 |
| 前端测试 | 测试阶段 | 原型设计 | 前端测试报告 |

## 3. 阶段 1：需求分析与设计

### 3.1 阶段目标

- ✅ 明确俊哥需求和产品目标
- ✅ 完成产品设计和原型
- ✅ 完成 6 大架构设计
- ✅ 输出完整 PRD 文档

### 3.2 工作流程

```
俊哥需求 → 产品设计师（PRD+ 原型） → 架构师（6 大架构评审） → 项目经理（优先级 + 计划） → 交付开发
```

### 3.3 各角色任务

#### 产品设计师

| 序号 | 任务 | 输入 | 输出 | 完成标准 | 预计时长 |
|------|------|------|------|----------|----------|
| 1 | 需求分析 | 俊哥需求 | 需求清单 | 需求明确、无歧义 | 1-2 天 |
| 2 | 功能设计 | 需求清单 | 功能清单（P0/P1/P2） | 优先级明确 | 1-2 天 |
| 3 | 用户流程设计 | 功能清单 | 用户流程图 | 学生/家长流程清晰 | 1 天 |
| 4 | 原型设计 | 功能清单 | 原型（Figma/墨刀） | 高保真 + 交互说明 | 2-3 天 |
| 5 | PRD 编写 | 以上所有 | PRD 文档 | 功能/流程/原型完整 | 2-3 天 |
| 6 | 设计评审 | 架构师意见 | PRD 修订版 | 通过评审 | 1 天 |

**输出文档**：
- `PRD_vX.X.md`（主文档）
- `user-flow.md`（用户流程图）
- `prototype.md`（原型设计）
- `feature-list.md`（功能清单）

---

#### 架构师

| 序号 | 任务 | 输入 | 输出 | 完成标准 | 预计时长 |
|------|------|------|------|----------|----------|
| 1 | PRD 评审 | PRD 文档 | 评审意见 | 识别技术风险 | 1 天 |
| 2 | 业务架构设计 | PRD+ 评审意见 | 业务流程图 | 业务边界清晰 | 1 天 |
| 3 | 应用架构设计 | PRD | 架构拓扑图 | 模块拆分合理 | 1 天 |
| 4 | 数据架构设计 | PRD | ER 图 + 数据流图 | 数据模型完整 | 2 天 |
| 5 | 技术架构设计 | PRD | 技术选型报告 | 选型合理、有备选 | 1 天 |
| 6 | 安全架构设计 | PRD | 安全评审报告 | 识别安全风险 | 1 天 |
| 7 | AI 架构设计 | PRD | AI 架构方案 | 模型路由清晰 | 1 天 |
| 8 | 架构决策记录 | 以上所有 | ADR 文档 | 每个决策有记录 | 持续 |
| 9 | 补充 PRD | 架构文档 | PRD 架构章节 | 与 PRD 整合 | 1 天 |

**输出文档**：
- `architecture-review-report.md`（架构评审报告）
- `architecture-risks.md`（风险清单）
- `adr-XXX.md`（架构决策记录，每个决策一个）
- PRD 文档架构章节

---

#### 项目经理

| 序号 | 任务 | 输入 | 输出 | 完成标准 | 预计时长 |
|------|------|------|------|----------|----------|
| 1 | 确认优先级 | PRD+ 架构文档 | 优先级清单 | P0/P1/P2 明确 | 0.5 天 |
| 2 | 制定项目计划 | 优先级清单 | 项目计划表 | 里程碑清晰 | 0.5 天 |
| 3 | 任务分配 | 项目计划 | 任务清单 | 责任到人 | 0.5 天 |
| 4 | 阶段验收 | 所有文档 | 验收报告 | 通过质量门禁 | 0.5 天 |

**输出文档**：
- `project-plan.md`（项目计划）
- `task-assignment.md`（任务分配）
- `phase1-acceptance.md`（阶段验收报告）

## 5. 阶段 3：测试验证

### 5.1 阶段目标

- ✅ API 测试通过（100% 接口覆盖）
- ✅ 前端测试通过（组件+E2E）
- ✅ 浏览器 E2E 测试通过
- ✅ 所有 P0/P1 Bug 已修复

### 5.2 工作流程

```
fullstack（提交测试） → qa（测试计划→API 测试→前端测试→E2E） → fullstack（Bug 修复） → 项目助理（归档）
```

### 5.3 各角色任务

#### 测试工程师（QA）

| 序号 | 任务 | 输入 | 输出 | 完成标准 | 预计时长 |
|------|------|------|------|----------|----------|
| 1 | 测试计划 | PRD+API 文档 | 测试计划 | 测试范围/用例/时间明确 | 0.5 天 |
| 2 | 测试用例编写 | 测试计划 | 测试用例 | 覆盖所有功能 | 1-2 天 |
| 3 | API 测试 | API 文档 | API 测试报告 | 100% 接口覆盖 | 1-2 天 |
| 4 | 前端测试 | 原型设计 | 前端测试报告 | 组件测试通过 | 1-2 天 |
| 5 | 浏览器 E2E | 用户流程 | E2E 测试报告 | 完整流程通过 | 1-2 天 |
| 6 | Bug 跟踪 | 测试结果 | Bug 清单 | P0/P1/P2 分级 | 持续 |
| 7 | 回归验证 | 修复代码 | 回归报告 | 所有 Bug 已修复 | 1 天 |

**输出文档**：
- `test-plan.md`（测试计划）
- `test-report.md`（测试报告）
- `bug-list.md`（Bug 清单）
- 测试截图（`docs/screenshots/`）

---

#### 全栈工程师

| 序号 | 任务 | 输入 | 输出 | 完成标准 | 预计时长 |

### 7.4 GitHub 同步规范

**同步频率**：每 30 分钟自动推送

**同步脚本**：`scripts/sync-github.js`

**同步内容**：
- ✅ 所有源代码（backend/ + frontend/ + mobile/）
- ✅ 所有文档（docs/ + project/v1-prd/docs/）
- ✅ 配置文件（.env.example, package.json, prisma/schema.prisma）
- ✅ 测试报告和截图

**不同步内容**（.gitignore）：
- ❌ node_modules/
- ❌ .env（敏感配置）
- ❌ *.db（本地数据库）
- ❌ uploads/（用户上传文件）
- ❌ 日志文件

**同步流程**：
```bash
# 1. 检查变更
git status

# 2. 添加所有文件
git add -A

# 3. 提交
git commit -m "auto sync: $(date '+%Y-%m-%d %H:%M:%S')"

# 4. 推送
git push origin main

# 5. 记录日志
echo "$(date): Synced to GitHub" >> docs/quality/github-sync-log.md
```
